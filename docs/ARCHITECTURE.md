# System Architecture

## 1. High-Level Diagram Description
The system follows a standard client-server architecture within a Turborepo monorepo.
- **Client (Mobile/Web)**: A Next.js application (App Router) serving all user roles. It uses local storage/IndexedDB for offline capabilities.
- **API Layer**: A FastAPI (Python) service exposing REST endpoints.
- **Database**: PostgreSQL with SQLAlchemy ORM and Alembic for migrations.
- **Shared Code**: UI components and logic shared via internal workspace packages.

## 2. Technology Stack
- **Monorepo**: Turborepo, pnpm
- **Frontend**: Next.js, Tailwind CSS, shadcn/ui
- **Backend**: FastAPI, SQLAlchemy, Alembic
- **Database**: PostgreSQL

## 3. Structural Components
### `apps/web`
- The single Next.js application.
- Responsive design: Mobile-optimized for Sales/Delivery; Desktop-optimized for Admin/Manufacturing.
- **Offline Strategy**: Uses a local action queue. Mutations (e.g., "Create Order") are stored locally when offline and replayed to the API when online.

### `apps/api`
- Python FastAPI service.
- Stateless REST API.
- Handles authentication, business logic, and database interactions.

### `packages/`
- **`ui`**: Shared React components (shadcn/ui), icons, and layout primitives.
- **`themes`**: Design tokens and global styles.
- **`utils`**: Shared helper functions (validators, formatters).

## 4. Offline Sync Strategy
- **Read**: Data required for offline work (Client lists, Product catalogue, consolidated Price tiers) is cached locally on standard login/sync.
- **Write**: Actions performed offline (e.g., submitting an order) are persisted to a robust local store (e.g., IndexedDB/TanStack Query persistence).
- **Sync**: A background worker detects connectivity and processes the outbound queue.
- **Idempotency**: Every offline action includes a client-generated UUID to ensure actions are not duplicated if retried.

## 5. Role-Based Access Strategy
- **Frontend**: Route guards and UI element hiding based on the signed-in user's role claim.
- **Backend**: Strict permission checks on every endpoint. An Admin API token cannot be generated by a Sales user.
- **Data Segregation**:
  - Sales users only fetch *their* orders.
  - Delivery users only fetch *assigned* deliveries.
  - Admins fetch *all* data.
